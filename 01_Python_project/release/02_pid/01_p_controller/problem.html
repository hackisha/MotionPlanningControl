P Controller — 비례 제어기 구현

Chapter 2 · PID / 01_p_controller  |  오차에 비례하는 제어 입력을 생성하는 가장 기본적인 피드백 제어기

1차원 점질량 시스템이 목표 위치로 수렴하도록 비례(P) 제어기를 설계해 보세요.

1문제 상황

차량이 차로 중앙(Y = 0)에서 약간 벗어난 위치(y₀ = 1.0 m)에서 출발합니다. 제어기는 매 시간 스텝마다 제어 입력 u를 출력하고, 시스템(plant)은 그 입력을 받아 위치를 갱신합니다. 목표는 가능한 한 빠르고 안정적으로 y = 0으로 수렴시키는 것입니다.

P 제어기는 메모리 없는(memoryless) 제어기입니다 — 매 호출에서 현재 오차만 보고 출력을 결정하므로 내부 상태를 저장할 필요가 없습니다. 이후 PD, PID 로 확장되는 흐름의 출발점입니다.

2주어진 환경 (Plant)

시뮬레이션 plant는 plant.py의 Plant 클래스(감쇠 점질량)입니다. 매 스텝 동역학: 

a  ← (u − damping · v) / m
v  ← v + a · dt
y  ← y + v · dt
  • 기본 파라미터: dt=0.1 s, y0=1.0 m, damping=1.0, m=1.0
  • 외란 없음 — P 제어만으로 수렴 가능한 환경
  • 테스트 게인: kp=2.0, 시뮬 30초

3구현 과제

p_controller.py의 PController.step() 메소드를 구현하세요. 클래스 골격과 시그니처는 이미 주어져 있습니다. 

class PController:
    def __init__(self, kp: float): ...

    def step(self, reference: float, measure: float) -> float:
        """한 스텝의 제어 입력 u 를 반환."""
인자 의미
reference 목표값 (이 과제에서는 0.0)
measure 현재 측정값 — plant의 현재 위치 y
반환 제어 입력 u — plant의 step(u)에 전달

게인·파라미터 튜닝

이 과제에서 구현하는 것은 알고리즘 로직 입니다 — 라이브러리 코드의 시그니처는 고정이고 본문(# TODO)만 채웁니다. 게인·파라미터의 실제 값 은 코드가 아니라 실행 파일에서 정하며, 세 군데의 역할이 다릅니다.

  • 라이브러리 코드 (구현 대상) — 알고리즘만 구현. 게인·파라미터는 매개변수로 받기만 하고 값은 담지 않음.
  • producer (record_gen.py / demo.py) — 게인·파라미터 값을 정하는 곳. release 기본값은 0 이라 그대로 실행하면 응답이 없거나 오류가 납니다 → 직접 값을 채워 실행하며 응답 변화를 비교하세요 (자유롭게 변경 OK).
  • test_*.py — 합격 기준 검증용 값. 변경 금지.

이 문제에서 바꾸는 줄 — record_gen.py 의 아래 줄. release 기본값은 0 이니 여기에 값을 채우세요:

controller = PController(kp=0.0)

4실행 방법

모든 명령은 git 저장소 루트에서 실행합니다.

구현 파일

01_Python_project/release/02_pid/01_p_controller/p_controller.py

테스트 — 합격 기준 검증

uv run pytest 01_Python_project/release/02_pid/01_p_controller/ -v

시나리오 실행 → record.json 생성 + Rerun viewer 자동 실행

uv run python 01_Python_project/release/02_pid/01_p_controller/record_gen.py

record_gen.py 안의 kp 값을 직접 채워 응답 변화를 비교하세요. release 기본값은 0이라 그대로 실행하면 응답이 나오지 않습니다. JSON만 만들고 viewer를 띄우지 않으려면 --no-viewer를 붙입니다.

챕터 전체 viewer로 재생

uv run python 01_Python_project/release/02_pid/simulator_pid.py 01_Python_project/release/02_pid/01_p_controller/

디버그 신호 — 심화 분석 (선택)

record_gen.py 의 dbg.add(...) 블록에는 debug1/2/3 주석 placeholder 가 준비돼 있습니다. 주석을 풀고 분석하고 싶은 값·식(예: 추적 오차, 내부 상태)을 채우면 그 신호가 record 에 함께 저장됩니다 — 신호 추가·삭제·수정은 이 dbg.add() 호출의 kwarg 한 줄로 끝납니다.

디버그 신호는 viewer 기본 화면에는 나타나지 않습니다 — Rerun viewer 좌측 entity 패널에서 /debug/<이름> 항목을 골라 TimeSeriesView 를 직접 추가하면 시계열로 볼 수 있습니다. 구현이 막힐 때 내부 신호를 들여다보는 디버깅 도구로 활용하세요.

5합격 기준 pytest

알고리즘 형태는 자유 — 동작 명세(behavioral spec)만 검증합니다.

  1. 폐루프 추적 오차 — y0=1.0, target=0.0, kp=2.0, 30초 시뮬 후 tail 평균 |오차| < 0.05, peak |오차| < 1.2

6힌트

  • 오차는 reference − measure 순서입니다. 반대로 쓰면 양의 피드백이 되어 발산합니다.
  • P 제어기는 메모리리스 — __init__에서 이전 측정값을 저장할 필요가 없습니다.
  • kp를 너무 크게 잡으면 dt=0.1 명시적 오일러에서 수치 발산이 생길 수 있습니다.
